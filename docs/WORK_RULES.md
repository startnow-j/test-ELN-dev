## 〇、定期回顾机制（重要）

### 0.1 为什么需要定期回顾？

在长会话中，AI 可能会因为上下文过长而遗忘早期阅读的详细规则。为确保规则持续生效，需要建立回顾机制。

### 0.2 回顾触发点

| 触发时机 | 回顾内容 | 回顾方式 |
|----------|----------|----------|
| 会话开始时 | WORK_RULES.md + PENDING_TASKS.md | 完整阅读 |
| 重大改动前 | RULES_QUICK_REF.md | 快速回顾 |
| 模块完成时 | RULES_QUICK_REF.md | 快速回顾 |
| 保存进度前 | RULES_QUICK_REF.md | 快速回顾 |
| 对话超过 50 轮 | RULES_QUICK_REF.md | 用户提醒或AI主动回顾 |

### 0.3 速查文件

`RULES_QUICK_REF.md` 是极简版规则速查表，包含：
- 核心检查清单
- 模块依赖关系图
- 关键文档位置
- 强制规则提醒

### 0.4 用户提醒指令

当发现 AI 偏离规则时，可使用以下指令：
- **"回顾规则"** → AI 重新阅读 RULES_QUICK_REF.md
- **"检查清单"** → AI 输出当前操作的检查清单
- **"你现在应该做什么"** → AI 回顾当前阶段应执行的规则

## 一、会话启动规则

### 1.1 文档一致性审核（必做）

每次新会话启动开发时，必须执行以下步骤：

```
┌─────────────────────────────────────────────────────────────┐
│                    会话启动检查清单                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  □ 1. 阅读 README.md 了解文档结构                            │
│  □ 2. 阅读 PENDING_TASKS.md 了解当前进度                     │
│  □ 3. 阅读 DEV_PLAN_v3.3.md 了解开发计划                     │
│  □ 4. 阅读 worklog.md 了解最近开发记录                       │
│  □ 5. 对照系统实际状态，确认文档与代码一致性                   │
│  □ 6. 如发现不一致，先更新文档再开始开发                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 一致性检查要点

| 检查项 | 检查方式 | 发现不一致时 |
|--------|----------|--------------|
| 功能完成状态 | 运行系统验证 | 更新 PENDING_TASKS.md |
| API端点 | 检查路由文件 | 更新对应模块文档 |
| 数据库模型 | 检查 prisma/schema.prisma | 更新模块文档中的数据模型章节 |
| 用户角色权限 | 测试各角色功能 | 更新 USER_MANAGEMENT_MODULE.md |

## 二、重大改动评估规则

### 2.1 触发评估的条件

以下情况需要进行影响评估：

| 触发条件 | 示例 |
|----------|------|
| 修改数据库模型 | 添加/删除字段、修改枚举值 |
| 修改API接口 | 更改请求/响应格式、更改权限要求 |
| 修改权限逻辑 | 更改角色权限、添加新权限检查 |
| 修改核心流程 | 更改审核流程、状态流转逻辑 |
| 添加新模块 | 新增独立功能模块 |
| 修改公共组件 | 修改被多处引用的组件 |

### 2.2 影响评估流程

```
┌─────────────────────────────────────────────────────────────┐
│                    影响评估流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 识别改动范围                                            │
│     ↓                                                       │
│  2. 阅读相关模块文档                                        │
│     - 必读：SYSTEM_FEATURES_v3.3.md（系统架构）              │
│     - 必读：直接相关模块文档                                 │
│     - 选读：间接相关模块文档                                 │
│     ↓                                                       │
│  3. 分析潜在影响                                            │
│     - 数据流向影响                                          │
│     - 权限影响                                              │
│     - 界面影响                                              │
│     - 已有功能影响                                          │
│     ↓                                                       │
│  4. 输出影响分析报告                                        │
│     - 改动内容概述                                          │
│     - 影响模块列表                                          │
│     - 风险评估                                              │
│     - 建议方案                                              │
│     ↓                                                       │
│  5. 与用户讨论确认                                          │
│     ↓                                                       │
│  6. 确认后启动开发                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 模块依赖关系图

```
用户管理 (USER_MANAGEMENT)
    ↓ 权限控制
项目管理 (PROJECT) ←───┐
    ↓ 关联            │
实验记录 (EXPERIMENT)  │
    ↓ 触发            │
审核管理 (REVIEW) ─────┘ 项目归档锁定
    ↓ 关联
文件管理 (FILE_MANAGEMENT)
    
我的任务 (MY_TASKS) ← 聚合多个模块数据
```

**依赖说明**：
- 修改用户管理模块 → 检查所有模块的权限逻辑
- 修改项目管理模块 → 检查实验记录、审核管理
- 修改实验记录模块 → 检查审核管理、文件管理、我的任务
- 修改审核管理模块 → 检查项目管理（归档锁定）、我的任务

### 4.2 保存流程

```
┌─────────────────────────────────────────────────────────────┐
│                    进度保存流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 工作总结                                                │
│     - 当日完成的功能                                        │

│     - 待解决的问题                                          │

│     - 运行 bun run lint 检查代码质量                        │

│  4. Git 提交                                                │
│     - git add .                                             │
│     - git commit -m "描述性提交信息"                        │

│                                                             │

│                                                             │

```

### 4.3 提交信息格式

```
<type>: <subject>

[optional body]

type 类型：
- feat: 新功能
- fix: 修复bug
- docs: 文档更新
- refactor: 重构
- style: 代码格式调整
- chore: 构建/工具变动

示例：
feat: 项目管理模块全局视角切换功能
fix: 修复项目基本信息编辑无法保存的问题
docs: 更新BACKUP_RESTORE文档，添加版本标签管理章节
```

## 五、效率提升规则

### 5.1 开发前检查规则

**目的**：避免重复开发，提高代码复用

| 检查项 | 检查位置 | 说明 |
|--------|----------|------|
| 组件是否已存在 | src/components/ui/ | shadcn/ui 组件优先使用 |
| 工具函数是否已存在 | src/lib/ | 避免重复编写相同功能 |
| Hook是否已存在 | src/hooks/ | 复用已有状态逻辑 |
| API是否已存在 | src/app/api/ | 扩展而非新建 |

**目的**：便于追溯历史，快速定位问题

### 5.5 问题解决记录规则

**目的**：积累经验，避免重复踩坑

在 worklog.md 中记录问题解决方案：

```markdown
## 问题解决记录

### 问题：项目基本信息编辑无法保存
**日期**：2025-02-28
**现象**：编辑开始日期、结束日期、描述后保存无效
**原因**：后端API使用 Object.assign 导致未提供的字段被覆盖为 undefined
**解决**：使用展开运算符只更新提供的字段
**代码示例**：
\`\`\`typescript
// 错误方式
Object.assign(existingProject, data)

// 正确方式
const updatedProject = await db.project.update({
  where: { id },
  data: {
    ...(data.startDate !== undefined && { startDate: data.startDate }),
    ...(data.expectedEndDate !== undefined && { expectedEndDate: data.expectedEndDate }),
    ...(data.description !== undefined && { description: data.description }),
  }
})
\`\`\`
```

## 六、风险预防规则

### 6.1 重大改动前备份

```
重大改动前执行：
1. 确认当前代码已提交到 Git
2. 创建备份标签：git tag -a backup-pre-xxx -m "备份说明"
3. 推送标签到远程
```

### 6.2 数据库变更谨慎处理

```
数据库变更规则：
1. 修改 schema.prisma 前先备份 db 目录
2. 运行 bun run db:push 后验证数据完整性
3. 如有数据迁移需求，先与用户确认
```

### 6.3 关键功能测试

```
改动完成后必须测试：
1. 多角色权限测试（使用 TEST_ACCOUNTS.md 中的账户）
2. 关键业务流程测试（创建→提交→审核）
3. 边界条件测试（空值、权限不足等）
```
